A comparison of three-dimensional rotation formalisms for least-squares and Bayesian inverse kinematics
¶

Appendix B
¶

Ben Serrien, Klevis Aliaj, Todd Pataky

In this appendix, we demonstrate the use of the pybik package for doing IK simulations. See the main paper for performance of the different methods and Appendix A for details on the mathematics and statistics. It requires the following packages to be installed: numpy, scipy and pymc2. The first two come pre-installed with python distributions like Anaconda. The last can be installed from the instructions given here: https://pymc-devs.github.io/pymc/ The pybik package was developed and tested on the following system: Python 3.7.7 (Anaconda 1.9.12), Numpy 1.19.1, Scipy 1.5.2, PyMC 2.3.8. The Anaconda environment file can be found in the repository. It was tested on Linux, Mac and Windows???

Set-up¶

Import the packages and set a seed for the RNG for replication purposes.

In [1]:
import numpy as np
import pybik
np.random.seed( 1 )

We set up the intial marker positions with known coordinates. In this example, we use a square marker cluster of 100$\times$100 mm² located at a distance of 50 mm from the origin. The marker coordinates are entered in a MarkerCluster object (r0).

In [2]:
r0 = np.array([[100.,0.,50.],[-100.,0.,50.],[0.,100.,50.],[0.,-100.,50.]])
r0 = pybik.MarkerCluster( r0 )

We simulate a random rotation (r) that we treat as the 'true' rotation. We only specify the rotation magnitude (theta), the rotation direction is random. The final coordinates (r1) are then calculated by applying the rotation to the initial coordinates.

In [3]:
theta = np.radians(45)                                 # rotation magnitude
r     = pybik.random.rotation3( theta )                # rotation object
r1    = pybik.MarkerCluster( r.apply( r0.toarray() ) ) # final coordinates

Here we print the different $SO(3)$ representations of the true rotation (for comparison with estimated rotations):

Now we add i.i.d. uncorrelated and isotropic Gaussian noise with standard deviation sigma to the true marker coordinates. The noisy "measured" coordinates (r0n, r1n) are used for IK.

In [5]:
sigma = 2.  # SD of the noise (in mm)
r0n   = r0.add_noise(sigma)
r1n   = r1.add_noise(sigma)

The noisy marker coordinates are entered in a IK mechanism object. The true value of sigma is entered as well which is used for initiating the MCMC algorithm in BIK.

In [6]:
model = pybik.mechanisms.rotation3(r0n, r1n, sigma)

Why the (absolute) difference between between $SO(3)$ parameterizations is not a proper metric for the estimation error?¶

Comparing the different IK solutions above to the true rotation, it is clear that the (absolute) difference between the true and estimated representations, is not a good error metric. For example, we print here the rotation vector representation of the true rotation, estimated rotation, error rotation, and difference between the true and estimated rotation. The latter two do not match in sign and magnitude. This is shown in the figure below (arrowheads don't touch). The metric from Huynh (2009) calculates the smallest angle between the rotation vectors and is always a proper metric.
